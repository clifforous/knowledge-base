## Facts

- A solution design for insights exists at `30_designs/ascent-rivals/eventun-post-match-insights-solution-design.md`.

## Assumptions

- Service-side implementation comes before full game-client integration.
- Eventun remains responsible for insight selection, ranking, confidence, policy, suppression, and factual status.
- The game client should not synthesize kudos from empty responses, promote secondary insights to primary, or collapse timeout, failed, unavailable, and no-insight states into one internal state.
- Progression/goals testing remains a secondary active task unless it blocks or informs the insights work.

## Review Checkpoint

- Service-side implementation should preserve the design boundary: Eventun selects insights; the game client renders localized facts.
- No candidate should be shown only to fill an insight slot.
- Timeout is not the same as no-insight; failed, unavailable, pending, timeout, and no-insight need distinct internal handling.
- Telemetry additions should stay compact and support phase-1 insight quality without becoming a full item recommender.

## Risks

- Insights can regress into shallow recommendations if weak coaching is allowed to fill empty slots.
- Economy and loadout insights can become wrong if they ignore heat number, mode, or valid early saving.
- Service work may need coordinated Eventun, protobuf, database, and game telemetry changes before client integration is meaningful.
