## Goal

Stabilize the game-client post-match recommendations experience and run a basic progression/goals-system verification pass.

## Facts

- Current primary work is fixing a recommendations bug in the game client.
- The desired no-good-recommendations behavior is a kudo rather than a low-quality filler recommendation.
- A return-to-lobby bug is causing player entries to show as grey after a match.
- Progression/goals needs a basic test pass.

## Assumptions

- Recommendation selection and ranking remain backend-owned. The client should not compute recommendation logic or re-rank recommendations locally.
- A no-good-recommendations kudo is a presentation fallback for suppressed or empty recommendation results, not a new recommendation category unless implementation review proves the API needs one.
- The first progression/goals test pass can be basic and manual unless an existing automated test harness is already available.

## Primary Tasks

- [ ] Reproduce the game-client recommendations bug and capture the failing state.
- [ ] Identify whether the defect is in client request wiring, response parsing, async state handling, UI presentation, or backend data returned to the client.
- [ ] Fix the recommendation display flow without moving backend-owned ranking or suppression logic into the client.
- [ ] Add the kudo fallback for the case where no recommendation is available or no recommendation clears the quality threshold.
- [ ] Verify the fallback does not mask transport errors, pending responses, or malformed recommendation payloads.
- [ ] Test the fixed recommendation flow with at least one normal recommendation result and one no-good-recommendations result.

## Return-To-Lobby Bug

- [ ] Reproduce the post-match return-to-lobby flow where player entries appear grey.
- [ ] Determine whether the grey entries represent stale readiness, disconnected presence, team assignment, player identity, or UI styling state.
- [ ] Fix the state refresh or presentation path so returned lobby player entries display the correct active state.
- [ ] Verify the fix across at least one completed-match return and one fresh lobby entry path.

## Progression / Goals Test Pass

- [ ] Confirm the basic logged-in player progression flow is reachable.
- [ ] Test active goals or challenges display for a logged-in player.
- [ ] Test post-match progression summary behavior for a completed match when available.
- [ ] Confirm pending progression-summary behavior is handled clearly if goal evaluation has not completed yet.
- [ ] Record any mismatches between implementation behavior and the progression design.

## Review Checkpoint

- Validate that the client distinguishes these states clearly: recommendation ready, no good recommendation, pending, failed/error, and unavailable data.
- Validate that post-match lobby return restores player roster presentation without carrying stale match-end visual state.
- Validate that no local client behavior contradicts the recommendation design's suppression rule: do not return filler.
- If progression/goals behavior differs from the current design, update the relevant knowledge-base document after confirming the implementation behavior is intentional.

## Risks

- Treating an empty recommendation list as an error could hide the intended no-good-recommendations path.
- Treating every empty or failed response as a kudo could hide real service failures.
- Progression/goals testing may require backend seed data, completed match data, or a known test account to produce meaningful results.
