# Record the verified pre-stream routing requirement discovered while implementing the public gauntlet detail route

## website-v2-coder-20260723:D001 — finding

- Timestamp: 2026-07-24T00:08:42.360061944Z
- Type: finding
- State: pending
- Statement: Under Next.js 16 Cache Components, Website V2 dynamic detail routes that require authoritative pre-stream HTTP status must resolve canonical identity and dependency availability before the partial-prerender shell streams. For the gauntlet detail route, a bounded server-side proxy preflight owns malformed/NotFound 404 classification and sanitized dependency 503 classification; page rendering then consumes the separately cached normalized public projection.
- Rationale: The approved failure model requires actual HTTP 404 and 5xx responses, not merely error UI inside a streamed 200 response. This is a reusable runtime constraint for dynamic Website V2 detail routes using Cache Components.
- Evidence: Production-mode fixture verification showed that a Cache Components page resolving notFound or a dependency error beneath Suspense can already have emitted the partial-prerender shell and therefore return HTTP 200. The implemented src/proxy.ts canonical-detail preflight produced actual shared 404 responses for malformed/NotFound identities and an actual sanitized 503 with Retry for dependency failure, while valid detail remained 200.

## website-v2-coder-20260723:D002 — correction

- Timestamp: 2026-07-24T00:47:50.041256883Z
- Type: correction
- State: pending
- Statement: Website V2 must validate public StageRun runNumber as a positive one-based value and render it unchanged. Only zero-based stage and match identities receive a human-facing +1 conversion.
- Rationale: StageRun runNumber is already a visitor-ready positive ordinal, unlike zero-based stage and match API identities. Preserving it prevents a systematic off-by-one display error and rejects impossible zero-valued contract fixtures.
- Evidence: Eventun run creation stores RunNumber as the previous maximum plus one, so the first persisted StageRun is 1. The initial Website detail implementation accepted zero and added one during rendering, which displayed persisted run 1 as Run 2.
- Destinations: ascent-rivals/initiatives/website-v2/pages/gauntlet-detail.md; ascent-rivals/initiatives/website-v2/README.md

## website-v2-coder-20260723:D003 — correction

- Timestamp: 2026-07-24T00:47:57.678931369Z
- Type: correction
- State: pending
- Statement: Website V2 public detail normalization must preserve recognized occurrence timing basis. Authored-duration ranges may render normally; stage-window estimates must be labeled as estimated, and unknown or unspecified timing basis must not be presented as exact.
- Rationale: The range remains useful schedule information, but visitors must be able to distinguish an estimated stage window from an authored duration. Unknown timing-basis enums must continue to fail closed rather than silently appearing exact.
- Evidence: The deployed public detail contract distinguishes PUBLIC_GAUNTLET_OCCURRENCE_TIMING_BASIS_AUTHORED_DURATION from PUBLIC_GAUNTLET_OCCURRENCE_TIMING_BASIS_STAGE_WINDOW_ESTIMATE. The initial Website normalizer retained start/end timestamps but discarded timingBasis, causing estimated stage ends to use the same unlabeled range presentation as authored qualifier durations.
