# Ascent Rivals Walletless Player Accounting - Future Work
## Executive Summary
The existing tournament accounting contract introduced a privacy-preserving accounting pattern where sensitive details are committed to Merkle trees and selectively disclosed when needed. All blockchain interaction is centralized in a backend service, and players never need a wallet for the core flow. Winners may optionally link a Midnight wallet address to claim a small Dust bonus for participating in a tournament.
This proposal explores extending those ideas from tournaments to player-held balances of Arc, our current in-game currency. The goal is to back Arc with an on-chain currency while keeping chain interaction completely optional for players. Players should be able to authorize sensitive actions using an existing game platform identity (Steam, Epic, and similar providers), rather than directly signing a transaction with a Midnight wallet.
The proposal is intentionally framed as future work. It outlines mechanisms that can be adopted incrementally and combined over time.
## Proposed Work
### On-Chain Player Accounts with Balance Sheet Accounting
One area of work is to create on-chain player accounts that track Arc balances through a balance sheet model. Each player account maintains an available balance. Credits, debits, transfers, and withdrawals update the available balance in a way that the contract can enforce “no negative balances” and avoid double-spend style failure modes.
Balance change entries are defined from a player identity reference, an amount, an action type (credit, debit, transfer, withdrawal), a salt, and a metadata note or reference (for example: match id, tournament id, or store order id). The entry is hashed into an append-only structure and the player’s balance is updated as part of the same operation. Transfers and withdrawals are treated as atomic operations.
This design keeps the currency representation flexible. The core goal is that balances and receipts are auditable and privacy-preserving, not that Arc must take a specific token form. Arc may still exist in normal self-custody accounts; the player account balance sheet is an additional lane that provides walletless usability for players who do not opt into managing their own account.
### Reserve backing
The service must hold sufficient funds in reserve to back the total liabilities recorded on the balance sheet. Credits, debits, and account-to-account transfers do not require moving funds in or out of reserve; they require that total reserve holdings remain greater than or equal to the total recorded player liabilities. Withdrawals reduce player liabilities and result in an actual payout from reserve to the player’s chosen destination.
### Game Platform Authorization for Walletless On-Chain Actions
A second area of work is to let players authorize actions performed by a Midnight backend service on their behalf using a third-party platform identity. Players sign into Ascent using one or more platform authentication services (Steam, Epic, PlayStation, and so on). These are linked to a single identity in our system. The goal is to use the platform authorization state at the time of request and produce an auditable trail connecting an on-chain action to a user authorization event that was valid at that time.
The proposed mechanism has two artifacts: validation receipts and spend authorization tokens.
### Validation receipts
When a player requests an identity-gated action (for example: a withdrawal or transfer), the backend validates the player’s platform authorization state and produces a validation receipt. The validation receipt is a structured record that commits to a hash of the platform authorization evidence captured at the time of the request, a hash of the action parameters the player approved, and timestamps with validity window information.
Validation receipts are appended off-chain to an append-only Merkle tree, and the backend periodically commits the current Merkle root on-chain. This provides a public audit anchor without placing sensitive platform tokens on-chain.
### Spend authorization tokens
For each identity-gated request, the backend issues a spend authorization token that is narrowly scoped to the requested action and bounded by a short validity window. The spend authorization token references the corresponding validation receipt and binds authorization to the exact action parameters.
One-time use is enforced by consuming a marker inside the contract. Each spend authorization token includes a unique nonce, and the contract derives a nullifier from that nonce (and optionally the player identity reference and purpose). The contract rejects any token whose nullifier has already been consumed, and consumes the nullifier as part of the same transaction that performs the authorized action.
### Auditing and disclosure
A player may request proof that they authorized a transaction. The backend should be able to produce an audit chain that links the on-chain transaction reference to the spend authorization token that authorized it, the corresponding validation receipt, and a Merkle inclusion proof showing that the validation receipt was included in the append-only tree rooted by an on-chain commitment. Subject to privacy policy, the backend should also be able to show the stored snapshot data needed to demonstrate that the platform authorization was valid at the time of the request.
### General flow
A player signs into the web application using a platform identity provider and requests a sensitive action. The backend validates the platform authorization state and records a validation receipt that commits to both the platform evidence and the action parameters the player approved. The backend then issues a spend authorization token scoped to that request and submits the on-chain transaction through the Midnight intermediary. The contract verifies the token is valid, unexpired, and not replayed before executing the authorized action.
### Combined Hybrid Approach
These areas of work can be combined into a single hybrid that provides both continuous accounting and auditable authorization. The balance sheet defines where balances live and how credits, debits, transfers, and withdrawals are tracked. The game platform authorization layer defines how a player can provide “final say” for sensitive operations without a wallet.
Initially, in-game credits and debits would flow through the balance sheet, while withdrawals and transfers require a validation receipt and a spend authorization token. Over time, the system can ratchet upward by requiring explicit authorization for a wider set of actions, up to and including selected debits if the product and threat model justify it.
## Out of Scope
This proposal intentionally does not specify the final on-chain representation of Arc, including token standard, bridging, market integrations, or a final custody design. The backing asset may not even be Arc; it may be another currency with Arc represented as an accounting unit.
This proposal does not attempt to define a universal identity protocol that other ecosystems must adopt. It narrowly defines a way to link an on-chain transaction to a particular game platform authorization event in a way that is auditable and time-bounded.
## System Context
Ascent: Rivals consists of a game client, a web application, and a backend that centralizes all interaction with Midnight. The backend intermediary remains the sole on-chain writer and is responsible for submitting contract calls, operating proof generation infrastructure, and maintaining the off-chain Merkle trees used for audit anchoring and selective disclosure.
In Compact, “ledger” refers to the contract’s replicated on-chain state. Contracts can store structured on-chain state using ledger declarations, including sets, maps, and bounded Merkle trees declared with a fixed maximum depth (and therefore bounded capacity). This proposal treats on-chain contract state as the authoritative accounting surface for the player balance sheet and as the enforcement point for one-time use authorization markers.
External platform identity systems (Steam, Epic, and others) participate indirectly by supplying authorization assertions that are validated by the backend and committed into validation receipts. Settlement rails remain backend responsibilities; the chain records accounting commitments and receipt references rather than implementing every settlement primitive on-chain. Players primarily interact through the game client and the web application. Wallets remain optional and are only required if a player chooses a self-custody path for Arc.
